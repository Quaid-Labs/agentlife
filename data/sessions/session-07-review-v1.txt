TURN 10 (SQL Injection Discovery)
TAG: normal | TOKENS: 87/100 | GENERATION TIME: 8.5s

AI ASSISTANT:
  Hey, while I was updating the search endpoint for the redesign, I
  noticed something I should flag.

  The search function has a SQL injection vulnerability. Look at this
  line in `server.js`:

  ```javascript
  // BEFORE (vulnerable):
  const recipes = db.prepare(
    "SELECT * FROM recipes WHERE name LIKE '%" + q + "%' OR ingredients LIKE '%" + q + "%'"
  ).all();
  ```

  That `q` value goes directly into the SQL string without sanitization.
  Someone could pass a crafted search query and mess with your database.
  All the other endpoints use parameterized queries correctly — this one
  just slipped through.

  Here's the fix:

  ```javascript
  // AFTER (safe):
  app.get('/api/search', (req, res) => {
    const { q } = req.query;
    if (!q) return res.json([]);
    const recipes = db.prepare(
      "SELECT * FROM recipes WHERE name LIKE ? OR ingredients LIKE ?"
    ).all(`%${q}%`, `%${q}%`);
    res.json(recipes);
  });
  ```

  Now the search term goes through prepared statement parameters — same
  pattern as all the other endpoints.

MAYA:
  wait what? that's been broken this whole time?

  like someone could have messed with my database?

ANALYSIS: The SQL injection fix. Agent discovers the bug from session 3
while working on the redesign. Explains the vulnerability clearly —
string concatenation vs parameterized queries. Maya's reaction is
surprise and mild alarm. "This whole time" emphasizes the time gap
(session 3 to session 7, about a week).
