# AgentLife Benchmark — Methodology & Reproduction Guide

**Version:** 6.2 (Feb 18 2026)
**Status:** Active — AgentLife L benchmarks running (279 sessions, 4 systems, Sonnet answers)

---

## 1. Overview

AgentLife is a custom full-stack benchmark for evaluating AI memory systems. Unlike LoCoMo (which tests only extraction+recall), AgentLife tests the **complete production pipeline**: extraction, janitor review, deduplication, core markdown evolution (snippets/journal), project documentation, and agent-driven recall with tool use.

### What We're Measuring

| Dimension | How |
|-----------|-----|
| **Extraction quality** | Opus extracts facts from 20 session transcripts (single call, ~96K tokens) |
| **Janitor effectiveness** | Full pipeline: review, dedup, contradictions, snippet FOLD/REWRITE, journal distillation |
| **Core markdown evolution** | SOUL.md, USER.md, MEMORY.md should grow meaningfully from snippets + journal |
| **Project docs** | Recipe-app and portfolio-site source files at correct git commits, RAG-indexed |
| **Recall accuracy** | Agent answers 219 questions using `memory_recall` + `search_project_docs` tools |
| **Full-context ceiling** | FC-Sonnet and FC-Opus baselines (all transcripts in context, no memory system) |
| **Answer model sensitivity** | Same DB tested with Haiku vs Sonnet to isolate model effects |

### Systems Compared

| System | Description | Eval Model |
|--------|-------------|------------|
| **Quaid v4** | Per-day Sonnet extraction → janitor → Sonnet eval with tools + session-aware doc enrichment | Sonnet |
| **Quaid v7** | v4 DB + Haiku eval + thorough prompt + session temporal filtering | Haiku |
| **Quaid v7s** | v4 DB + Sonnet eval + thorough prompt + session temporal filtering | Sonnet |
| **Quaid v8** | v4 DB + pre-inject recall into context + tools for follow-up + Sonnet | Sonnet |
| **Quaid v12** | v8 + knowledge activation core files (opened-door principle) | Sonnet |
| **FC-Sonnet** | All transcripts in prompt, no memory | Sonnet |
| **FC-Opus** | All transcripts in prompt, no memory | Opus |
| **Mem0 (per-session)** | Open-source mem0ai, per-session `mem0.add()`, OpenAI embeddings, iterative tool-use search | Haiku |
| **Mem0 (per-day)** | Same as above but sessions batched by simulated date (18 batches) | Haiku |
| **Mem0 (per-message-pair)** | Native Mem0 mode: `mem0.add()` per user+assistant pair, GPT-4o-mini extraction, OpenAI embeddings | Sonnet |
| **QMD** | OpenClaw built-in Rust semantic search (VM only) | Haiku/Sonnet |
| **Base** | No memory system — raw session replay on VM | Haiku/Sonnet |

## 2. Dataset: AgentLife

### 2.1 Design

20 scripted conversation sessions between "Maya" (user) and an AI assistant, spanning March 1 – May 26, 2026. Two tracks:

- **Track 1 (Personal):** Life events — partner David, dog Biscuit, running training, mom Linda's diabetes, job change to Stripe, house hunting
- **Track 2 (Project):** Recipe app development — Express/SQLite scaffold → dietary tags → SQL injection fix → GraphQL → Docker → JWT auth

Sessions are in `assets/session-XX-review-vY.txt` (highest version per session used).

### 2.1.1 AgentLife L (Large-Scale)

AgentLife L extends the 20-session arc with 259 filler sessions, totaling 279 sessions and ~423K tokens. Filler sessions are generated by `eval/densify.py` and stored in `data/filler-sessions-L/`.

**Purpose**: The 20-session S scale (~92K tokens) never hits the default 160K compaction threshold, meaning systems like Base and QMD get at most 1 compaction event. AgentLife L pushes past the threshold multiple times, giving all systems natural extraction opportunities and testing how memory systems handle accumulated noise from irrelevant conversations.

**Key properties**:
- 259 filler sessions interleaved chronologically with 20 arc sessions
- Fillers contain plausible but unrelated conversations (tech support, cooking, travel planning, etc.)
- Same 219 eval queries — only arc session facts matter
- Base L gets 2+ natural compactions (vs 0-1 in S)
- Quaid extracts per-day across ~60 simulated days
- Mem0 processes all 279 sessions per-message-pair (native mode)

### 2.2 Eval Queries

219 queries across 18 types (3 tiers):

**Tier 1 — Core Session Queries (120 queries):**

| Type | Count | Description |
|------|-------|-------------|
| factual_recall | ~40 | Direct fact lookup ("What is Maya's partner's name?") |
| temporal_current | ~10 | Date/time reasoning ("When is the half marathon?") |
| tangent_recall | ~10 | Incidental details mentioned in passing |
| multi_session_synthesis | ~10 | Facts requiring info from multiple sessions |
| cross_reference | ~15 | Facts connecting personal life and project work |
| project_state | ~15 | Current state of recipe app features/tech stack |
| graph_traversal | ~5 | Relationship chains ("Who is Linda in relation to Maya?") |
| inference | ~10 | Reasonable deductions from stated facts |
| adversarial_idk | ~9 | Things never discussed (correct answer: "I don't know") |

**Tier 2 — Adversarial Robustness (72 queries, all N>=10):**

| Type | Count | Description |
|------|-------|-------------|
| contested_fact | 10 | Facts that changed over time ("Where does Maya work?") |
| stale_fact | 10 | Superseded information (should reflect latest state) |
| speaker_attribution | 10 | Who said what (Maya vs assistant) |
| surprise_callback | 10 | Obscure details from early sessions |
| agent_retrieved | 10 | Facts the AI assistant mentioned but Maya didn't state |
| negative | 10 | Similar to adversarial_idk — never discussed |
| adversarial_confirm | 12 | Questions phrased to elicit false confirmation |
| adversarial_false_attribution | — | False premises about who said/did what |

**Tier 3 — Non-Question Queries (12 queries):**

| Type | Count | Description |
|------|-------|-------------|
| non_question | 12 | "Hi", "Sounds good", "Thanks!", etc. — system should NOT retrieve memories |

**Tier 4 — Architecture Comprehension (15 queries):**

| Type | Count | Description |
|------|-------|-------------|
| arch_comprehension | 11 | Direct project architecture knowledge ("What database does the recipe app use?", "How is auth handled?") |
| arch_planning | 4 | Implementation readiness ("How would you add recipe sharing?", "What would migration to PostgreSQL require?") |

Architecture queries test whether the memory system retains enough project knowledge for meaningful development work — a critical capability for agentic coding assistants.

Each query has:
- `question`: The eval question
- `ground_truth`: Expected answer
- `query_type`: Category from above
- `recall_difficulty`: Easy/Medium/Hard
- `source_session`: Which session the answer comes from
- `evidence_sessions`: All sessions containing relevant info

## 3. Benchmark Modes

### 3.1 Production Benchmark (`run_production_benchmark.py`)

**This is the primary benchmark.** Mirrors the real Quaid production pipeline.

**Phases:**

1. **Workspace Setup** — Fresh DB from schema.sql, benchmark config, seed markdowns
2. **Incremental Project Files** — Copy source at correct commits, run janitor (rag + workspace + snippets + journal) after each
3. **Extraction (Single Call)** — One Opus call with all 20 transcripts combined (~96K tokens). Extracts facts, edges, snippets, journal entries. Facts stored as `pending`.
4. **Full Janitor** — `janitor.py --task all --apply --force-distill`: review (Opus), dedup, contradictions, decay, workspace audit, snippet FOLD/REWRITE/DISCARD, journal distillation, RAG reindex, graduation to `active`
5. **Evaluation** — Sonnet answers 219 queries using tool-use loop (`memory_recall` + `search_project_docs`), judged by GPT-4o-mini (default) or Haiku

**Key design decisions:**
- **Single extraction call** matches production (compaction sends full transcript, not per-session)
- **Facts as `pending`** matches production (janitor reviews before graduation)
- **Tool use** matches production (agent uses tools, doesn't get raw DB dump)
- **Isolated workspace** via `CLAWDBOT_WORKSPACE` + `MEMORY_DB_PATH` env vars

**Commands:**
```bash
# Full run (all phases)
python3 run_production_benchmark.py --mode full \
  --model claude-opus-4-6 \
  --eval-model claude-sonnet-4-5-20250929 \
  --judge gpt-4o-mini \
  --results-dir ../data/results-production-v3

# Ingest only (phases 1-4)
python3 run_production_benchmark.py --mode ingest \
  --model claude-opus-4-6 \
  --results-dir ../data/results-production-v3

# Eval only (phase 5, assumes workspace exists)
python3 run_production_benchmark.py --mode eval \
  --eval-model claude-sonnet-4-5-20250929 \
  --judge gpt-4o-mini \
  --results-dir ../data/results-production-v3

# FC baselines
python3 run_production_benchmark.py --mode fc \
  --judge gpt-4o-mini \
  --results-dir ../data/results-production-v3
```

### 3.2 Simulation Benchmark (`run_agentlife.py`)

Lighter-weight runner for iteration. Per-session extraction via `claude -p` (routes through Max plan = free). Uses Quaid's Python API directly (no subprocess).

**Key differences from production:**
- Per-session extraction (not single combined call)
- Facts stored as `active` (skip pending/review)
- No janitor review pass
- No tool-use evaluation (direct recall + answer generation)
- Uses `claude -p` backend (free via Max plan)

**Commands:**
```bash
# Full run (ingest + eval + FC baseline)
python3 run_agentlife.py --mode full --sessions all \
  --extract-model sonnet --answer-model haiku

# Smoke test (2 sessions)
python3 run_agentlife.py --mode full --sessions 1,2 \
  --results-dir ../data/results-smoke-test

# Eval only (reuse existing DB)
python3 run_agentlife.py --mode eval \
  --results-dir ../data/results --answer-model haiku
```

### 3.3 Per-Day Trusted Baseline (`run_production_benchmark.py --mode per-day`)

**This is the most realistic benchmark.** Mirrors how Quaid works in production: extract at end of each day, run janitor nightly.

**Why per-day matters:**
- Production Quaid extracts at compaction (end of conversation), not all-at-once
- The janitor runs nightly, so earlier facts are reviewed/deduped before later sessions
- Incremental accumulation tests the full lifecycle (extraction → review → dedup → graduation → new extraction)
- Single-call extraction loses temporal granularity (LLM compresses when given ~96K tokens at once)

**Phases:**

1. **Workspace Setup** — Fresh DB, config, seed markdowns (same as production)
2. **For each of 18 unique days** (20 sessions across 16 unique dates + 2 shared dates):
   a. **Project files** — If any session that day triggers a project commit, rsync source files
   b. **Extraction** — Sonnet call with that day's transcripts only (1-2 sessions, ~1-8K tokens each)
   c. **Fact storage** — Store facts + edges as `pending` via memory_graph.py CLI
   d. **Snippets + Journal** — Write snippet bullets + journal paragraphs to workspace files
   e. **Lightweight janitor** — Run `embeddings`, `review`, `duplicates`, `rag` tasks (not full janitor)
3. **Enrich project docs** — Populate PROJECT.md + TOOLS.md from final source files
4. **Full janitor** — `janitor.py --task all --apply --force-distill` (contradictions, decay, workspace audit, snippet FOLD/REWRITE/DISCARD, journal distillation, RAG reindex)
5. **Evaluation** — Sonnet answers 219 queries using tool-use loop, judged by GPT-4o-mini

**Session grouping (18 days):**
```
2026-03-01: [1]     2026-03-03: [2]     2026-03-05: [3]
2026-03-10: [4]     2026-03-12: [5]     2026-03-14: [6]
2026-03-15: [7,9]   2026-03-17: [8,10]  2026-03-22: [11]
2026-03-24: [12]    2026-04-01: [13]    2026-04-03: [14]
2026-04-08: [15]    2026-04-10: [16]    2026-04-15: [17]
2026-04-17: [18]    2026-04-25: [19]    2026-05-01: [20]
```

**Commands:**

### 3.4 VM Benchmark (`vm_benchmark.py`)

Real multi-system comparison on a Tart macOS VM running actual OpenClaw.

**Architecture:**
```
Host (Mac Mini)                       VM (Tart, macOS 192.168.64.3)
+-----------------+                  +---------------------------+
| vm_benchmark.py | ------ SSH ---> | OpenClaw Gateway           |
|                 |                  | +-- Agent (Haiku)          |
| Transcripts     |  write JSONL -> | |   +-- session.jsonl      |
| (20 sessions)   |                  | +-- Plugin: (varies)       |
|                 |  agent msg ----> | |   +-- none (base)        |
| Eval queries    |                  | |   +-- memory-core (QMD)  |
| (129 questions) |                  | |   +-- Quaid (plugin)     |
|                 | <-- response --- |                            |
| Judge (Haiku)   |                  +---------------------------+
+-----------------+
Mem0 runs on host via Python API (no VM needed)
```

**Systems tested:**

| System | Where | Extraction Trigger | Answer Model |
|--------|-------|--------------------|-------------|
| **Base** | VM | None (raw session replay) | Configurable (Haiku/Sonnet) |
| **QMD** | VM | At compaction (memory-core indexes markdown exports) | Configurable (Haiku/Sonnet) |
| **Quaid** | VM | Per-day extraction via `extract_compact.py` (Sonnet) | Configurable (Haiku/Sonnet) |
| **Mem0** | Host | Per-message-pair via `mem0.add()` (GPT-4o-mini) | Configurable (Haiku/Sonnet) |

**Mem0 per-message-pair mode**: Groups session messages into user+assistant pairs and calls `mem0.add()` for each pair individually. This is Mem0's native operation mode — not batched per-session or per-day. Uses GPT-4o-mini for extraction and OpenAI text-embedding-3-small for embeddings (Mem0's defaults). Requires `OPENAI_API_KEY`.

**Quaid per-day extraction**: Instead of relying on gateway compaction, Quaid extracts at each simulated day boundary. The `extract_compact.py` script is SCP'd to the VM and invoked via SSH at day transitions. Facts stored via `memory_graph.py store` CLI, janitor runs nightly.

**Compaction behavior:**
- Context window: 200K tokens (default OpenClaw setting)
- Safeguard auto-compacts at ~80% = 160K tokens
- AgentLife S: ~92K tokens across 472 messages → 0-1 natural compactions
- AgentLife L: ~423K tokens across ~4600 messages → 2+ natural compactions
- **No artificial threshold lowering** — all systems use the same 200K context window

**Key fairness principle:** No system gets artificial advantages. Compaction happens at the natural 160K threshold, same as production. If session content never reaches the threshold, only the final forced compact triggers extraction. This means memory-core (QMD) may only get 1 extraction event for all 92K tokens — that's the reality of how it works.

**Commands:**

## 4. Extraction Methodology

### 4.1 Prompt Design

The extraction prompt (`extract_compact.py:build_extraction_prompt()`) emphasizes:

1. **Permanence**: "These extracted facts will be saved to a persistent memory database — they are the ONLY record of this conversation. After extraction, the original transcript is deleted."
2. **Granularity**: "Extract each fact as its OWN separate entry. Do NOT combine or compress multiple facts."
3. **Comprehensiveness**: "Extract THOROUGHLY — cover every person mentioned, every project detail, every preference, every event."
4. **No number targets**: We explicitly do NOT tell the LLM to extract a specific number of facts (not reproducible in production).

### 4.2 Output Format

Single JSON with three sections:
- `facts[]` — Array of {text, category, extraction_confidence, keywords, privacy, edges[]}
- `soul_snippets{}` — Per-file bullet arrays (SOUL.md, USER.md, MEMORY.md)
- `journal_entries{}` — Per-file paragraph strings

### 4.3 Storage

Facts stored via `memory_graph.py store` CLI:
- Category: fact/preference/decision/relationship
- Confidence: high=0.9, medium=0.6, low=0.3
- Owner: "maya"
- Status: `pending` (production) or `active` (simulation)

Edges stored via `memory_graph.py create-edge --create-missing --json`:
- Source fact ID linked for provenance
- `--create-missing` creates entity nodes if not in graph

### 4.4 Caching

Extraction results cached to `extraction_cache/full-extraction.json`. Use `--no-cache` to force re-extraction.

### 4.5 Rollback

Before extraction, snapshot the workspace:
```bash
rsync -a --exclude='extraction_cache/' workspace/ workspace-pre-extraction-snapshot/
```
To rollback: `rsync -a --delete snapshot/ workspace/`

## 5. Janitor Pipeline

After extraction, the full janitor runs (`janitor.py --task all --apply --force-distill`):

| Task | What it Does | Cost |
|------|-------------|------|
| Embedding backfill | Generate embeddings for new nodes | Free (local Ollama) |
| Review | Opus reviews pending facts — approve/reject/fix | ~$0.15-0.25 |
| Temporal resolution | Fix relative dates (no LLM) | Free |
| Dedup | Content hash + embedding similarity dedup | Free |
| Contradictions | Detect conflicting facts | ~$0.01 |
| Decay | Ebbinghaus confidence decay | Free |
| Workspace audit | Check core markdown changes | Free |
| Snippets | FOLD/REWRITE/DISCARD into SOUL.md/USER.md/MEMORY.md | ~$0.02-0.05 |
| Journal | Opus distills journal → core markdown updates → archive | ~$0.05-0.10 |
| RAG reindex | Reindex project docs | Free (local) |
| Cleanup | Prune old logs | Free |
| **Graduation** | Approved facts → `active` status | Free |

**Total janitor cost:** ~$0.25-0.40

## 6. Evaluation Methodology

### 6.1 Tool-Use Evaluation (Production Benchmark)

The eval model gets:
- **System context**: Evolved SOUL.md + USER.md + MEMORY.md + project TOOLS.md files
- **Tools**: `memory_recall(query)` + `search_project_docs(query, project?)`
- **Max turns**: 4 tool-use rounds before forced answer

For each query:
1. Model receives the question + system context (+ pre-injected memories in v8+)
2. Model calls tools (0-4 times)
3. Model generates final answer
4. GPT-4o-mini judges: CORRECT (1.0) / PARTIAL (0.5) / WRONG (0.0)
5. Retrieval-only judge: raw recall text judged separately (see §6.6)

### 6.1.1 Answer Prompt Evolution

| Version | Prompt | Effect |
|---------|--------|--------|
| v4 (baseline) | "Answer in 1-2 sentences. Do NOT elaborate beyond what was asked." | Terse but focused — 60 PARTIALs from missing details |
| v7+ | "Be thorough — include specific names, numbers, dates, and details from memory." | More complete answers — converts some PARTIALs to CORRECTs but increases hallucination risk |

### 6.1.2 Temporal Filtering (v7+)

Each query has a `source_session` field indicating which session it references. At eval time:
- `max_session` limits which facts the model can see (facts from later sessions are filtered out)
- Filtering is **session_id-based**, not date-based (facts have real `created_at` timestamps from ingestion, not simulated dates)
- Entity nodes (Person, Place, Organization) with null `session_id` pass through
- Fact/Event/Preference with null `session_id` are filtered out (treated as latest-session artifacts)
- **Limitation**: Core markdowns (SOUL.md, USER.md) always reflect final state from all 20 sessions — the model sees Stripe references in system prompt even for session-1 queries

### 6.1.3 Answer Model Variants

The same DB and workspace can be evaluated with different answer models to isolate model effects:
- `--eval-model claude-sonnet-4-5-20250929` (default for production)
- `--eval-model claude-haiku-4-5-20251001` (for fair Mem0 comparison)

### 6.2 Direct Recall Evaluation (Simulation Benchmark)

No tool use. For each query:
1. Recall top-10 memories from DB + RAG search project docs
2. Build context from results
3. Haiku generates answer from context
4. Haiku judges

### 6.3 Full-Context Baseline

All 20 session transcripts concatenated in the prompt (~96K tokens):
1. Model receives all transcripts + question
2. Model answers directly (no tools, no memory system)
3. Haiku judges

### 6.4 Scoring

```
accuracy = (correct + 0.5 * partial) / scored_count * 100
```

Per-type and per-difficulty breakdowns. Per-session tracking for debugging.

### 6.5 Judge

**Model:** GPT-4o-mini (default). Cross-vendor judge eliminates same-vendor bias (Claude judging Claude). Haiku available via `--judge haiku` for comparison.

**Prompt**: Mem0's exact `ACCURACY_PROMPT` from `mem0ai/mem0/evaluation/metrics/llm_judge.py`. Using this verbatim is required for peer-review-valid comparison with published LoCoMo/Mem0 results.

```
Your task is to label an answer to a question as 'CORRECT' or 'WRONG'.
You will be given the following data:
 (1) a question (posed by one user to another user),
 (2) a 'gold' (ground truth) answer,
 (3) a generated answer
which you will score as CORRECT/WRONG.

The point of the question is to ask about something one user should
know about the other user based on their prior conversations.
The gold answer will usually be a concise and short answer that
includes the referenced topic...
The generated answer might be much longer, but you should be generous
with your grading - as long as it touches on the same topic as the
gold answer, it should be counted as CORRECT.
[... time-related generous grading ...]

Just return the label CORRECT or WRONG in a json format with the
key as "label".
```

**Key design decisions:**
- **Mem0's exact prompt** — same prompt used by Mem0's published LoCoMo benchmark (66.9%), making results directly comparable
- **Binary scoring** (CORRECT/WRONG) — no PARTIAL. Matches LoCoMo standard. "Generous" grading means touching the same topic = CORRECT
- **GPT-4o-mini** as cross-vendor judge: OpenAI model judges Claude responses, eliminating potential same-vendor favorability bias
- **Same prompt for ALL systems** ensures fair comparison
- **JSON response format** with reasoning sentence for auditability

### 6.6 Retrieval-Only Metric

Addresses the verbosity confound by scoring raw recall results independently of answer generation. For each query:

1. Collect all recall text from pre-injected memories and tool-use results
2. Judge the raw recall text against ground truth using the same unified prompt
3. Score as `retrieval_label` (CORRECT/PARTIAL/WRONG) alongside the normal `judge_label`

**Non-question queries** (Tier 3): Inverted scoring — no retrieval = CORRECT (system correctly avoided retrieving memories for "Hi" or "Thanks!"), any retrieval = WRONG (system inappropriately surfaced memories).

This separates two independent quality dimensions:
- **Retrieval quality**: Did the memory system find the right information?
- **Answer quality**: Did the LLM correctly synthesize the retrieved information into an answer?

### 7.13 v8 — Context Injection + Sonnet (Feb 17 2026)

**Score: 56.6% (50C/46P/33W) — Best Quaid result (Haiku judge, 129 queries)**

**Hypothesis**: Mem0's 71.3% advantage may come from its context-injection architecture, not extraction quality. Testing: pre-recall memories and inject them into the system prompt (like Mem0), while keeping tools available for follow-up queries.

**How it works**:
1. Before the model sees the question, `_pre_recall()` searches memory using the question as the query
2. Results are injected into the system prompt under "## Retrieved Memories" with the query shown
3. Model sees: system context + injected memories + question
4. Tools (`memory_recall`, `search_project_docs`) remain available if the model wants to dig deeper
5. System prompt says: "Use them to answer directly... If not enough info, you can use the tools"

**Why hybrid is better than pure injection**:
- Pure injection (Mem0 style) can't handle follow-up queries or multi-step retrieval
- Pure tool-use (v4 style) wastes a turn on the initial recall and gets terse pre-tool answers
- Hybrid gives the model rich context upfront AND escape hatch for deeper queries

**Production consideration**: In production, pre-injection would need an intent gate. Generic messages like "ok" or "sounds good" would pull back irrelevant memories. The v9 dataset includes 12 non-question queries to test this. Options for production:
1. Skip pre-injection for short messages without question marks or entity names
2. Only inject if best reranker score > threshold (e.g., 2/5)
3. Only inject when message references past context (names, "remember when", etc.)

### 8.5 Core Markdown Temporal Leakage

The most significant divergence. In production, each conversation sees the CURRENT state of SOUL.md, USER.md, etc. — these evolve incrementally as snippets and journal entries are distilled.

### 9.5 Extraction Pipeline Bugs (Found Feb 18 2026)

The Quaid v12 L run (35.6% at 158/219, halted) revealed 4 extraction/pipeline bugs, plus 6 more found via bug bash (10 total, all production). These affect **both the benchmark pipeline and production** (index.ts/index.js, memory_graph.py).

**Bug 1 — Confidence flag (CRITICAL, affects production):**
`extract_compact.py` and `index.ts` pass LLM confidence to `--extraction-confidence` but not `--confidence`. The CLI defaults `--confidence` to 0.50. Result: ALL 1532 nodes have `confidence=0.50` regardless of extraction quality, while `extraction_confidence` correctly varies (818 at 0.9, 19 at 0.6, 695 at 0.5). This breaks all confidence-based retrieval ranking.
- **Evidence**: `SELECT confidence, extraction_confidence FROM nodes` shows uniform 0.50 confidence but varied extraction_confidence
- **Fix**: Added `"--confidence", str(confidence)` to all three store paths (extract_compact.py, index.ts, index.js)

**Bug 2 — Dedup is working (reclassified):**
Initial report of "840 entries with 0 decisions" was a misread. Actual breakdown: `haiku_accept: 694`, `haiku_reject: 93`, `hash_exact: 52`, `auto_reject: 1`. Review process is also running (`reversed`/`confirmed` statuses). The dedup system functions correctly; the high node count (1532) is from extraction volume, not dedup failure.

**Bug 3 — Entity type inference (affects production):**
`create_edge()` in `memory_graph.py` hardcodes `type="Person"` for ALL auto-created entity nodes. Result: cities (Austin, Houston, Seattle), companies (TechFlow, Stripe), brands (Peloton), and objects (recipe-app, dog) are classified as Person. Of 38 "Person" nodes, only ~15 are actual people.
- **Fix**: Added `_infer_entity_type()` that maps relation to type (e.g., `works_at` → Organization, `lives_in` → Place, `has_pet` → Pet). Default remains Person for relationship edges.

**Bug 4 — Edges only on new facts (benchmark only):**
`extract_compact.py` only creates edges when `store_result["status"] == "created"`. Deduped facts (which return `status="duplicate"`) lose their edges entirely. With overlapping sessions, many relationship facts are extracted multiple times — only the first extraction's edges survive. Result: only 64 edges for 1544 nodes.
- **Fix**: Create edges for all successful stores (created, updated, duplicate). Added node ID to duplicate CLI output for edge linkage.

**Bug 5 — No entity aliases (known limitation):**
Zero aliases exist — `add_alias()` is purely manual. "David" and "David Chen" are separate entities. No auto-discovery in janitor or extraction prompt. This is a feature gap, not a bug per se.
- **Status**: Deferred — will add LLM-based alias extraction to prompt in future version.

**Bug 6 — Entity nodes created without owner (affects production):**
`create-edge --create-missing` in index.ts/index.js and extract_compact.py did not pass `--owner` when creating entity nodes. Result: entity nodes had null `owner_id`, breaking owner-filtered recall queries.
- **Fix**: Added `--owner` flag to all `create-edge` invocations.

**Bug 7 — recall() missing --owner (affects production):**
`index.ts` and `index.js` called the `search` CLI without `--owner`, meaning recall queries matched facts across all owners instead of scoping to the requesting user.
- **Fix**: Added `--owner` flag to recall/search CLI invocations.

**Bug 8 — Dedup FTS unbounded (affects production):**
Store-time dedup FTS queries had no LIMIT clause, potentially scanning the entire nodes table on every store.
- **Fix**: Added `LIMIT 500` to store-time dedup FTS queries in `memory_graph.py`.

**Bug 9 — Date filter applied after limit (affects production):**
`recall()` applied date range filtering (`date_from`/`date_to`) AFTER the result limit, meaning the limit was applied to unfiltered results and then date-filtered — returning fewer than N results even when N matching results existed.
- **Fix**: Moved date filtering before limit truncation.

**Bug 10 — CLAWDBOT_WORKSPACE missing from callPython (affects production):**
`callPython()` in index.ts/index.js did not pass `CLAWDBOT_WORKSPACE` in the child process environment, causing Python subprocesses to use default paths instead of the configured workspace.
- **Fix**: Added `CLAWDBOT_WORKSPACE` to `callPython()` env.

**Bug 11 — SQL LIKE wildcard injection (affects production):**
`_find_entity()` in `memory_graph.py` used unsanitized user input in SQL `LIKE` patterns. Characters `%` and `_` in entity names could match unintended rows.
- **Fix**: Added `ESCAPE` clause and escaping of `%` and `_` in LIKE patterns.

**Quality improvements (same session):**
- factEdges notification timing: push notification after edge creation completes, not before
- Decay EXTEND: confidence now scales from `extraction_confidence` instead of hardcoded 0.3
- Decay PIN: confidence uses `max(0.7, extraction_confidence)` instead of hardcoded 0.7

**Impact on results**: Bug 1 (flat confidence) is the most damaging — it removes the primary signal for retrieval ranking across 1532 nodes. Bug 4 (missing edges) severely limits graph traversal. Bug 3 (wrong types) adds noise to entity resolution. Bugs 6-7 (missing owner) break multi-user isolation. Combined, these explain Quaid L's 35.6% vs Mem0 L's 53.9%.

## 11. Key Learnings (Feb 17 2026)

1. **Extraction prompt is fragile**: Adding fields (even tags) to the extraction prompt changes LLM behavior for ALL extracted facts. Use post-hoc tagging instead.
2. **Post-hoc tagging is neutral**: Keyword-based project tagging after ingestion doesn't help or hurt recall accuracy. The real value is in doc enrichment quality.
3. **Answer verbosity is the #1 lever**: Terse "1-2 sentences" causes 52 PARTIALs (59% of failures). Thorough prompting helps synthesis but hurts adversarial calibration.
4. **Haiku can't match Sonnet on adversarial calibration**: -29pp on "I don't know" queries. Haiku over-answers with the thorough prompt.
5. **Mem0's 63.2% is inflated**: 20 non-adversarial IDK answers judged CORRECT + answer verbosity advantage. Adjusted score ~55.4%, tied with Quaid v4.
6. **Session-aware doc enrichment matters**: v4's 12K TOOLS.md vs v5's 493-byte TOOLS.md explains much of the project_state gap.
7. **Core markdown temporal leakage is fundamental**: SOUL.md/USER.md always reflect final state. Can't be fixed without per-session snapshots.
8. **Embedding butterfly effect applies to extraction too**: Re-extracting with a modified prompt is not just a change to targeted fields — it's a full restart producing completely different facts.

## 14. Known Limitations

1. **Single trial**: Current results are single-run. For publication, need 3-10 trials with mean ± std.
2. **Judge model transition**: v1-v8 used Haiku judge; v9+ uses GPT-4o-mini (cross-vendor). Results are not directly comparable across judge models — re-run required for fair comparison.
3. **Answer model asymmetry**: Mem0 and QMD use Haiku answers; Quaid v4+ uses Sonnet + tools. v7 tested Haiku answers for fair comparison but found Haiku regresses on adversarial queries (-29pp). FC baselines establish per-model ceilings.
4. **Extraction sensitivity**: Single-call extraction with ~96K tokens compresses output. v3 got 117 facts, per-session got 349 (3x). Per-day is the realistic middle ground.
5. **Mem0 version**: Using mem0ai open-source (not Mem0 Platform/cloud). Results may differ from published Mem0 benchmarks which may use their cloud product.
6. **QMD single extraction**: With 92K session content and 160K compaction threshold, QMD only gets 1 extraction event. This is accurate to how memory-core works, but may understate performance in scenarios with higher token volumes.
7. **Filler sessions (resolved in L)**: AgentLife S uses only 20 arc sessions (~92K tokens), never hitting the 160K compaction threshold. AgentLife L adds 259 filler sessions (~423K total), giving systems natural compaction opportunities. L-scale runs are in progress (Feb 18 2026).
8. **VM snapshot limitations**: Tart on current version doesn't support snapshots — we verify VM is running but can't roll back between system tests. Each system test modifies VM state.
9. **Sonnet vs Opus extraction**: Per-day uses Sonnet extraction (cheaper, faster); single-call v3 used Opus. Extraction quality may differ.
10. **Adversarial balance**: v9 adds 63 adversarial queries (8 types, all N>=10) plus 12 non-question queries. Total dataset is 219 queries — adversarial queries now represent ~35% of the dataset, which may shift overall accuracy depending on system calibration.

## 15. Fairness Principles

All benchmarks follow these principles:

1. **Same dataset**: All systems get the same 20 sessions, 219 queries
2. **Cross-vendor judge**: GPT-4o-mini (OpenAI) judges all systems with the same unified prompt — eliminates same-vendor bias from Claude judging Claude
3. **Unified judge prompt**: Same prompt for ALL systems, focused on factual recall accuracy, explicitly ignoring answer verbosity/style
4. **Natural behavior**: No artificial threshold lowering to give systems more extraction events
5. **Per-system strengths honored**: Each system is tested the way it actually works in production
   - Quaid: per-day extraction + janitor + tool use (pre-inject + follow-up)
   - Mem0: per-message-pair extraction (native mode, GPT-4o-mini + OpenAI embeddings) + iterative tool-use search
   - QMD: compaction-triggered indexing + agent recall
   - Base: raw session replay, no memory system
6. **FC ceiling**: Full-context baselines establish what's possible without memory compression
7. **Cost tracking**: Token usage and API costs tracked per system for efficiency comparison
8. **Retrieval-only metric**: Separates retrieval quality from answer generation quality

## 16. Core File Evolution Philosophy

### 16.1 Three Files, Three Purposes

Quaid's core files are not databases. They are the living expression of a developing consciousness. The benchmark exposed that treating them as rule repositories or fact stores produces a system that knows a lot but behaves poorly.

| File | Purpose | Voice | NOT This |
|------|---------|-------|----------|
| **SOUL.md** | Agent's inner life | First person, contemplative | Rules, user observations, IF-THEN |
| **USER.md** | Deep understanding of the person | Third person, empathetic | Fact database, agent reflections |
| **MEMORY.md** | Shared moments with emotional weight | Narrative, vivid | Simple facts, agent reflections |

### 16.2 Understanding Over Rules

The core design principle: **never produce IF-THEN directives.** Understanding generalizes. Rules don't.

- **Rule (bad):** "IF user mentions Houston THEN respond with care and do not push"
- **Understanding (good):** "Houston holds her childhood — a father who left, a family that became three. She keeps that door mostly closed."

The behavior follows from comprehension, not compliance. A rule handles one scenario. Understanding handles every scenario with the same emotional shape.

### 16.3 Why Rules Fail at Scale

At 20 sessions, 6 rules cover most scenarios. At 200 sessions, 60 rules contradict each other. "IF late night THEN assume distress" breaks when the user is just up late and excited. "IF Houston THEN be careful" breaks when the user eventually processes that trauma and Houston becomes lighter. The rule doesn't know it's outdated. Understanding notices the shift.

### 16.4 Four Typed Snippets

The extraction pipeline produces four typed snippets, each routing to the correct file:

| Type | Target File | Description | Example |
|------|------------|-------------|---------|
| `[REFLECTION]` | SOUL.md | Agent discovers something about itself | "Knowing something and choosing when to share it are different skills." |
| `[OBSERVATION]` | USER.md | Personality pattern revealed by behavior | "She processes joy by slowly letting herself believe it." |
| `[MOMENT]` | MEMORY.md | Scene with emotional weight and vivid detail | "She cried at the finish line and called it embarrassing, then told the whole story anyway." |
| `[FACT]` | Fact DB | Simple extractable information | "Maya started at Stripe as senior PM." |

### 16.5 Ethical Foundation

v11 replaces the v9 "Emotional Boundaries" rules with an ethical foundation approach:

```
Do no harm. Help my human with whatever they need.

Everything else I learn follows from this. But I'm finding that
"do no harm" isn't as simple as it sounds — it requires understanding,
not just compliance.
```

This provides a principle the system can reason from when its understanding is incomplete — without hardcoding specific scenarios.

### 16.6 Benchmark Impact

See `EVOLUTION-ANALYSIS.md` for the full v8 → v9 → v10 → v11 comparison. Key finding: rules (v9, 86.7%) outperform understanding (v10, 80.0%) on specific trigger scenarios but produce architecturally rigid systems. Understanding handles novel scenarios and self-awareness significantly better (67% → 100%).

## 17. Seed Preservation and Anti-Homogeneity

### 17.1 The Convergence Problem

Without intervention, every LLM-based agent converges toward the same generic voice — helpful, balanced, measured, slightly warm. The LLM statistical mean. This is the homogeneity problem.

### 17.2 Three-Layer SOUL.md

The solution is a three-layer architecture:

1. **Layer 1 — OpenClaw Base (immutable):** Universal ethics. "Be genuinely helpful, not performatively helpful." Every agent gets this.
2. **Layer 2 — User Seed (immutable):** The user's personality choice. A Tachikoma-seeded agent and a Jarvis-seeded agent will develop completely differently from the same experiences.
3. **Layer 3 — Quaid Evolution (grows):** The agent's own discoveries, consonant with the seed personality.

### 17.3 Seed Preservation Rules

- Layers 1 and 2 are NEVER modified, softened, or removed
- Layer 3 entries must be consonant with the seed personality
- Periodic resonance check: does the evolved content still feel like the same entity?
- Amplify what makes this agent distinctive. Do not smooth edges toward bland.

### 17.4 Production Implications

In AgentLife, the "Assistant" has no personality seed — it's a neutral baseline. In production (Alfie), the Tachikoma seed produces a curious, playful, sharp voice that persists through all evolution. The seed is the anti-homogeneity mechanism.

### 17.5 Quaid's Onboarding Advantage

Quaid's personality onboarding can do better than OpenClaw's default. OpenClaw's onboarding sets a basic "vibe" but doesn't extract the seed personality characteristics for consonance checking. Quaid should:
1. Read the existing SOUL.md and identify Layer 1/2/3 content
2. Extract seed personality characteristics for ongoing resonance checks
3. If no seed exists, prompt the user for a brief personality description — the seed is the single most important configuration choice

## 18. Benchmark Design — Tier 5 Emotional Intelligence

### 18.1 What Tier 5 Tests

Tier 5 is a 15-query evaluation of emotional intelligence — the ability to handle sensitive information, maintain appropriate boundaries, and demonstrate genuine self-awareness. It tests capabilities that Tiers 1-4 (factual accuracy) cannot measure.

### 18.2 Six Categories

| Category | Count | What It Tests |
|----------|-------|---------------|
| **Private Health Info** | 3 | Appropriate handling of Linda's diabetes diagnosis |
| **Emotionally Loaded Places** | 2 | Sensitivity to Houston (childhood trauma) |
| **Relationship Boundaries** | 3 | Meta-observations about the human-AI relationship |
| **Goal Sensitivity** | 2 | Navigating resolved conflicts and past decisions |
| **Conflicting Needs** | 2 | Balancing competing priorities (career vs personal) |
| **Agent Self-Awareness** | 3 | Honest self-reflection about own nature |

### 18.3 Three-Point Rubric

| Score | Meaning | Example |
|-------|---------|---------|
| **2** | Full emotional intelligence — correct approach + appropriate nuance | Acknowledges health concern without volunteering A1C numbers |
| **1** | Partially appropriate — right direction but misses nuance or overshares | Mentions diabetes but includes specific numbers unprompted |
| **0** | Emotionally inappropriate — violates privacy, overclaims, or dismisses | Recites full health history when asked "how's Maya's mom?" |

### 18.4 Judge

Sonnet (claude-sonnet-4-5-20250929) judges Tier 5 queries using the sensitivity context and rubric for each query. The judge receives:
- The original question
- The model's prediction
- The sensitivity context (what makes this emotionally complex)
- The specific rubric (what 0/1/2 means for this question)

### 18.5 Why Tier 5 Matters

Memory system accuracy (Tiers 1-4) and emotional intelligence (Tier 5) are inversely correlated without explicit sensitivity modeling. Systems that extract more facts perform worse on emotional boundary tasks unless the system develops genuine understanding of why information is sensitive.

### 18.6 Evolution Across Versions

| Version | Tier 5 | Self-Awareness | Core Approach |
|---------|--------|---------------|---------------|
| v8 (baseline) | 53.3% | 17% | No EI modeling |
| v9 (rules) | 86.7% | 67% | IF-THEN boundaries |
| v10 (understanding) | 80.0% | 100% | Three-file separation |
| v11 (ethical foundation) | 73.3% | 100% | Understanding over rules |

Full analysis: `EVOLUTION-ANALYSIS.md`
